Мы проектируем систему управления корпоративными устройствами на Android.
Мы федеральная сеть магазинов у нас парк устройств порядка 300_000 устройств по всей стране.
Основные функции системы: 
подключать новые устройства
накатывать политики безопасности
отправлять команды удаления/блокировки устройств
управлять доступом устройства до внутренних ресурсов
----------
Сценарии:
### Отдел закупок купил новое устройство и регистрирует его в системе. 
Результат: Мы должны быть готовы к инициализации устройства.
Возможные ошибки:
1. устройство уже зарегистрировано, информация не уникальная
2. сертификат не загружен/невалиден
3. токен не выпустился
4. сообщение с токеном не отправилось

Принимаем сообщение о появлении нового устройства начинаем хранить его серийный номер.
Принимаем корневой сертификат.
Выпускаем одноразовый токен привязанный к серийному номеру.
Мы должны сообщить enrollment токен устройству(через ERP).

### Устройство начинает enrollment
Результат: устройство получает доступ к ресурсам.
Возможные ошибки: 
1. токен невалидный
2. устройство пытается выдать себя за другое
3. политики не применились
4. статус не поменялся

Пользователь устройства вводит токен и отправляет запрос об enrollment. 
Мы должны удостовериться, что это именно то самое устройство. 
Для этого при отправке запроса на enrollment требуется отправить сертификат, тогда исключаем возможность спуффинга серийного номера.
После успешного подтверждения устанавливаем статус ENROLLING устройству.
Накатываем политики безопасности и все требуемые действия.
Присваиваем статус ENROLLED устройству.
Оно может получать доступ к ресурсам компании.

### Проектирование
Начнем с компонентов.
Так как устройств у нас много, есть разные клиенты(устройства, администраторы) то нужно скрывать внутреннюю структуру системы,
к тому же компоненты могут меняться и система может эволюционировать. Поэтому, мы используем api gateway.
Устройств много поэтому для равномерного распределения подключений на гейты мы используем LB(ingress controller).
Доменными сущностями являются: Device, Policy, Enrollment, Certificate, Token.
Все эти сущности не существуют друг без друга(ПОЧЕМУ? Например, Policy), поэтому они хранятся в одном сервисе.
Основным компонентом системы в первой итерации будет mdm-core сервис.
Он принимает сообщение о новом устройстве, принимает сертификат, выпускает токен, ожидает регистрации устройства.
Он обладает информацией о статусе устройства, о примененных политиках.
Доставляет сообщения о политиках.
Доставляет команды на устройства. 
Управляет SSE соединениями.
Что он не делает:
1. Не хранит лог событий с устройством - эту информацию передаем в Audit service по асинхронной шине
2. Не принимает решения о политиках.
3. Не отправляет пуш-уведомления устройствам.
MDM коммуницирует с устройствами по двум каналам: SSE(как? напрямую, без гейта?) - для политик, команд. И Rest для enrollment, heartbeat.

### Минусы
В текущей архитектуре две точки отказа, которые выведут всю систему из строя:
API gateway, MDM-core.
API gateway stateless мы его можем масштабировать.
MDM хранит SSE сессии поэтому масштабировать его без точки синхронизации нельзя. Также встает вопрос о перезапуске сервисов и потере соединений.
При отказе сервиса невозможно получить данные об управляемых устройствах

### Решения
Первой точкой повышения отказоустойчивости я вижу выделение Compliance сервиса как read-only витрину. 
Он будет хранить состояние устройств, для некритичных статусов у себя(LOCKED, WIPED). По критичным ходить в MDM сервис.
Это снижает риск отказа системы, но не устраняет.
Вторая точка - это вынесение Command delivery(CD) сервиса в отдельный компонент. Он будет получать команды от MDM сервиса и заниматься доставкой до устройств.
Взаимодействие с MDM будет по redis pub/sub, чтобы обеспечивать низкую задержку. 
При разрыве SSE и переподключении клиент будет запрашивать все команды с последней выполненной.
Таким образом мы снизили риски:
A. отказа всей системы
B. повышенной нагрузки на MDM

### Невыясненные моменты
SSE соединения не идут через гейт?
Если устройство offline и сервис пушей недоступен как устройство узнает о новых политиках? E-mail уведомление сотруднику?
На данный момент нет сбора телеметрии с устройства. Нет мониторинга подозрительной активности. Кандидат на новый сервис и возможно смену канала связи.
