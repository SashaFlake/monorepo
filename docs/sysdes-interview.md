Мы проектируем систему управления корпоративными устройствами на Android.
Мы федеральная сеть магазинов у нас парк устройств порядка 300_000 устройств по всей стране. 
Они не подключаются единовременно, но в рабочие часы могут использоваться активно. Это могут быть сканеры товаров и прочее.
Система федеральная на всю Россию, из-за географии задержка может увеличиваться. Система должна быть готова к шардированию.
Основные функции системы: 
подключать новые устройства
накатывать политики безопасности
отправлять команды удаления/блокировки устройств
управлять доступом устройства до внутренних ресурсов
----------
Сценарии:
### Отдел закупок купил новое устройство и регистрирует его в системе. 
Результат: Мы должны быть готовы к инициализации устройства.
Возможные ошибки:
1. устройство уже зарегистрировано, информация не уникальная
2. сертификат не загружен/невалиден
3. токен не выпустился
4. сообщение с токеном не отправилось

Принимаем сообщение о появлении нового устройства начинаем хранить его серийный номер.
Принимаем корневой сертификат.
Выпускаем одноразовый токен привязанный к серийному номеру.
Мы должны сообщить enrollment токен устройству(через ERP).

### Устройство начинает enrollment
Результат: устройство получает доступ к ресурсам.
Возможные ошибки: 
1. токен невалидный
2. устройство пытается выдать себя за другое
3. политики не применились
4. статус не поменялся

Пользователь устройства вводит токен и отправляет запрос об enrollment. 
Мы должны удостовериться, что это именно то самое устройство. 
Для этого при отправке запроса на enrollment требуется подтвердить сертификат: отправить nonce,
в ответ получить signed nonce(число+сертификат), тогда исключаем возможность спуффинга серийного номера.
После успешного подтверждения устанавливаем статус ENROLLING устройству.
Накатываем политики безопасности и все требуемые действия.
Присваиваем статус ENROLLED устройству.
Оно может получать доступ к ресурсам компании.

### Проектирование
Начнем с компонентов.
Так как устройств у нас много, есть разные клиенты(устройства, администраторы) то нужно скрывать внутреннюю структуру системы,
к тому же компоненты могут меняться и система может эволюционировать. Поэтому, мы используем api gateway.
Устройств много поэтому для равномерного распределения подключений на гейты мы используем LB(ingress controller).
Доменными сущностями являются: Device, DeviceGroup, Policy, Enrollment, Certificate, Token.
Все эти сущности не существуют друг без друга, поэтому они хранятся в одном сервисе.
Основным компонентом системы в первой итерации будет mdm-core сервис.
Он принимает сообщение о новом устройстве, принимает сертификат, выпускает токен, ожидает регистрации устройства.
Он обладает информацией о статусе устройства, о примененных политиках.
Доставляет команды на устройства через механизм poll.
Устройство периодически запрашивает очередь команд. Интервал настраивается per DeviceGroup.

Что он не делает:
1. Не хранит лог событий с устройством - эту информацию передаем в Audit service по асинхронной шине
2. Не принимает решения о политиках.
3. Не отправляет пуш-уведомления устройствам.
MDM коммуницирует с устройствами по одному каналу — REST. Enrollment, poll команд, ACK — всё через REST.
SSE используется только для Admin UI (real-time dashboard оператора).

### Почему не SSE для устройств
Первоначально рассматривался SSE как push-канал для доставки команд на устройства.
От него отказались по трём причинам:
1. **Батарея.** Постоянное соединение держит радиомодуль активным — критично для сканеров и терминалов на батарее.
2. **Не помогает при краже.** WIPE/LOCK — главный срочный сценарий. Украденное устройство уже вне корпоративной сети, SSE-соединение уже разорвано. Команда всё равно ждёт в очереди.
3. **Сложность без выгоды.** 300k постоянных соединений требуют отдельного CDS, Redis pub/sub, sticky routing. Poll — stateless, горизонтально масштабируется бесплатно.
Compliance Service блокирует доступ к ресурсам немедленно при смене статуса — ещё до того, как агент заберёт команду. Задержка в пределах poll_interval приемлема.

### Минусы
В текущей архитектуре две точки отказа, которые выведут всю систему из строя:
API gateway, MDM-core.
API gateway stateless мы его можем масштабировать.
MDM Core stateless (не держит соединений) — масштабируется горизонтально без ограничений.
При отказе Core невозможно получить pending команды — устройства продолжают работать с применёнными политиками.
При отказе Core Compliance отвечает fail-closed — устройства не получают доступ к ресурсам. Это правильное поведение.

### Архитектурные решения
Первой точкой повышения отказоустойчивости я вижу выделение Compliance сервиса как read-only витрину. 
Он будет хранить состояние устройств: при негативных сценариях(WIPED, LOCKED, REVOKED) мы сразу возвращаем "Отказано", при отрицательном результате 
проверки soft-state - тоже отказ. Если hard-state && soft-state, то read запрос к MDM. Сервис stateless поэтому масштабируем.
Это снижает риск отказа системы, но не устраняет - при отказе MDM-core финального подтверждения не будет и никто не сможет залогиниться, 
что хорошо, потому что цена ошибочного пропуска выше чем отказа.

### DeviceGroup и poll_interval
Каждое устройство принадлежит группе. Группа определяет интервал poll.
Интервал — бизнес-параметр, настраивается без деплоя.

Примеры групп:
- critical (POS-терминалы) — 1 минута
- standard (сканеры склада) — 5 минут  
- low_power (редко активные) — 15 минут

При 300k устройств в группе standard с интервалом 5 минут: ~1000 RPS на /device/commands/pending.
Это управляемая нагрузка для stateless Core с индексом по device_id.
При poll устройство получает актуальный poll_interval в теле ответа — интервал можно менять на лету.
Heartbeat отдельным эндпоинтом не нужен: last_seen_at обновляется при каждом poll.

### Troubleshooting
Проблема: <em>у токена должен быть TTL. Какой срок знает только бизнес.</em>

Решение: <em>TTL enrollment токена - это бизнес параметр. Он может быть настраиваемым: от нескольких часов до нескольких дней.
Дефолтный 72 часа(день упаковка, день доставка, день на активацию)
Если у устройства недостаточно политик либо еще какая-то разрешимая проблема, то мы должны вернуть клиенту результат проверки из которой он может понять
в какое состояние требуется привести устройство. Например, если не все необходимые политики применены - говорим обнови политики. 
Клиент инициирует pull политик, для этого должен быть публичный endpoint(авторизация через сертификат).
В случае если enrollment прервался, то требуется ручной перезапуск. Отслеживание таких случаев - ответственность MDM сервиса.</em>
Проблема: 
<em>Как исключить спуффинг устройств</em>

Решение: <em>
Спуффинг устройства - это подмена устройства, либо утечка сертификата.
Signed nonce доказывает что устройство прямо сейчас владеет приватным ключом к этому сертификату.
Скопировать подпись нельзя — приватный ключ сгенерирован в Android Keystore и аппаратно не экспортируется.
MDM-core выдает nonce ключ если сертификат совпадает. Nonce ключ истекает по ttl.</em>

Проблема: <em>Если устройство offline оно не узнает о новых политиках мгновенно — не сможет подключиться сразу после обновления политики.</em>
Решение: <em>Если много политик пропущено устройство может быть скомпрометировано. Требуется реактивация либо другие ручные действия.  
E-mail уведомление сотруднику - kindly reminder. При следующем poll устройство получит команду APPLY_POLICY и применит актуальную политику.
Compliance Service тем временем ответит NON_COMPLIANT пока политика не применена.</em>

Проблема:
<em>География - от Калининграда, до Владивостока</em>

Решение: <em>Шардируем устройства по географическому принципу. Устройства ***никогда*** не меняют свое местоположение дальше своего региона просто так.
Логистика может быть слишком сложной и дорогой. Случаи редкие могут быть разрешены в ручном режиме.
Первый кандидат на шардирование база данных MDM-core</em>

### Пробелы в системе
На данный момент нет сбора телеметрии с устройства. Нет мониторинга подозрительной активности. Кандидат на новый сервис и возможно смену канала связи.

Не показана observability - вынесено за скоуп. Предполагается, что у нас есть distributed tracing, Prometheus, Grafana и т.д.
Сбор бизнес метрик. Должен быть data-pipeline который выгружает статистику с сервисов. Можно это сделать в графане как первая итерация,
но в дальнейшем это должен быть какой-то data platform.
Доставка push уведомлений: если нет доступа к глобальным сервисам, то как будем доставлять уведомления? Failback до смс? 
А если в устройстве нет сим-карты? Решение: reminder на самом устройстве.
